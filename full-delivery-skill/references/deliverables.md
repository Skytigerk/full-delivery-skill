# Deliverables Reference

## Default Deliverable Types

根据任务类型，至少交付以下之一；如果用户要求“完整交付”，优先交付两种以上：

1. **结果物**
   - 文件、脚本、配置、补丁、文档、表格、命令集
2. **结论物**
   - 最终结论、优先级判断、决策建议、风险结论
3. **证据物**
   - 命令输出摘要、测试结果、截图说明、路径列表、diff 摘要
4. **操作物**
   - 可复现步骤、启动命令、验证命令、回滚步骤

## Task → Deliverable Mapping

### 1. 代码/工程任务
最少交付：
- 修改后的文件路径
- 变更摘要
- 验证命令
- 已知风险

### 2. 研究/调研任务
最少交付：
- 结论摘要
- 关键信息点
- 来源/证据
- 建议下一步

### 3. 文档任务
最少交付：
- 完整文稿或可直接粘贴内容
- 结构说明
- 待补信息位

### 4. 排障任务
最少交付：
- 根因假设
- 已验证项
- 未验证项
- 下一步排查命令或修复动作

## Definition of Done Examples

### Example: 修一个 bug
- 能定位问题范围
- 有明确修改
- 有至少一条验证方法
- 有风险说明

### Example: 写一份方案
- 方案结构完整
- 关键假设写明
- 有执行步骤
- 有风险与备选方案

### Example: 做一次代码审查
- 列出问题
- 标出严重级别
- 给出修复建议或直接修改
- 给出是否可合并结论

## Blocked Delivery Pattern

如果被阻塞，仍然交付：

- **已完成**：已经做了什么
- **缺失**：缺什么权限/信息/环境
- **你需要提供**：最小补充项
- **下一步**：拿到信息后会怎么继续

## Response Style

- 先结果，后解释
- 优先给可执行内容，不堆废话
- 不要把“建议”伪装成“完成”
- 如果用户明确说“别偷懒”，优先减少铺垫，直接给交付包
